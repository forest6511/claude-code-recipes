# TDD開発サイクル

以下のRed-Green-Refactorサイクルに厳密に従ってください。

## 対象機能
$ARGUMENTS

## フェーズ1: Red（テストを書く）

1. 実装対象の仕様を明確にする
2. 失敗するテストを書く
3. テストを実行し、**必ず失敗することを確認する**
4. テスト結果を表示する

## フェーズ2: Green（最小限の実装）

1. テストを通す**最小限の**コードを書く
2. 過剰な実装をしない（テストが通るだけの最小コード）
3. テストを実行し、**全て通ることを確認する**
4. テスト結果を表示する

## フェーズ3: Refactor（リファクタリング）

1. コードの重複を排除する
2. 命名を改善する
3. テストを実行し、**依然として全て通ることを確認する**
4. テスト結果を表示する

## ルール

- 各フェーズの完了後、テスト実行結果を必ず表示する
- テストが通っていない状態で次のフェーズに進まない
- テストフレームワークは既存のプロジェクト設定に従う
- テストファイルの配置は既存の規約に合わせる
